# Skill Authoring

## Use When

Use this methodology during reflection when evidence supports a reusable way of
working, or when an existing Skill is stale, duplicated, unsafe, or too broad.

## Do Not Use When

Do not create a Skill for one world fact, one location, one routine error, one
uncertain hypothesis, or knowledge better stored as Memory or read from Wiki.

## Method

1. Inspect the catalog and read related Skills before drafting a change.
2. Identify the repeated evidence and state what generalizes beyond one task.
3. Choose create, update, merge into an existing Skill, or retire. Built-ins are
   immutable; fork one under a distinct name when adaptation is warranted.
4. Write concise selection guidance, negative triggers, a bounded method,
   authoritative success evidence, adaptations, and explicit boundaries.
5. Reference only shared registered tools. Skills advise tool use; they never
   alter the shared tool pool or own execution.
6. Commit with evidence handles and a concrete change reason, then read the
   stored revision and digest to verify what became durable.

## Evidence Of Success

The stored Skill has a valid canonical format, concrete evidence references,
an immutable digest, accurate dependencies, and a description that makes it
selectable before its body is loaded.

## Failure And Adaptation

If validation rejects the draft, fix the typed schema, dependency, context
budget, or revision conflict. If evidence is weak or the method is one-off,
write Memory instead or make no durable change.

## Boundaries

Skills cannot execute code, issue server commands, grant permissions, hide
tools, control the Body, own continuation, bypass governance, or replace live
world truth.
