# Recovery And Continuation

## Use When

Use this methodology after preemption, death, restart reconciliation, mobility
failure, or a tool result that leaves the admitted objective incomplete.

## Do Not Use When

Do not use it to manufacture continuation after terminal impossibility, to
repeat a stale action, or to turn routine errors into durable memory.

## Method

1. Re-read the task, checkpoint, and authoritative Body state. Treat pre-crash
   or preemption-era action results as stale until world truth confirms them.
2. Classify the typed blocker: transport, owner preemption, mobility, approach,
   missing prerequisite, governance refusal, or terminal impossibility.
3. Search durable memory only when a stable place, obstacle, or prior method
   could materially change the next attempt. Re-observe mutable world facts.
4. Resume the objective after a Body reflex completes. Do not create a competing
   action while another owner controls the Body.
5. Select a materially different approach only when evidence supports it.

## Evidence Of Success

Use fresh Body state, inventory delta, action terminal events, and explicit task
checkpoints. Recovery is complete only when the original objective has current
terminal evidence or an honest typed blocker.

## Failure And Adaptation

If no viable next action exists, checkpoint or yield with the exact blocker.
Do not extend an internal retry loop or rename repeated failure as progress.

## Boundaries

Recovery never bypasses player-block governance, authoritative terminal truth,
the single Body owner, or the admitted task lifetime.
